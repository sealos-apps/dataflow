# Bind Sealos sessions to database instance UID

For every Sealos-supported database engine, DataFlow uses the Kubernetes UID of the owning KubeBlocks `Cluster`, scoped by Kubernetes cluster, namespace, API group, kind, and resource name, as the authoritative **Database Instance Identity**. The DataFlow backend resolves this identity directly from Kubernetes with the Sealos kubeconfig; dbprovider supplies target metadata but does not assert instance identity. A reusable resource name, endpoint, credential hash, Secret UID, or `resourceVersion` does not represent the instance lifecycle, so a session associated with a different instance UID must not be reused after a same-name instance replacement. On an explicit Sealos launch, a read-only GraphQL identity operation resolves the current UID to decide whether an unexpired session can be reused. A mismatch or a legacy session with no stored UID requires `BootstrapSealosSession`; bootstrap independently resolves the UID again, immediately revokes every server-side Sealos session bound to the replaced UID plus unverified legacy sessions for that same target, and then attempts database connectivity before storing the verified UID with the new session. If DataFlow cannot resolve the authoritative UID, it refuses to reuse or create the database session instead of falling back to names, endpoints, credentials, or another Kubernetes object's identity. An already-open page may report database errors after deletion but does not poll for identity changes or silently migrate its pending work to a replacement instance.
