# DataFlow — 产品需求文档（PRD）

> **范围**：本文件盘点 DataFlow 在当前版本**已实现的全部用户可感知能力**。SQL / MongoDB / Redis 三类数据源因功能差异较大，独立成节。
> **读者**：产品 / 业务 / 解决方案同学。
> **不包含**：技术实现（语言、框架、接口协议）、部署拓扑、未来路线图、Phased Rollout。
> **当前版本**：`0.1.0`

---

## 0. 产品定位

**DataFlow 是一个面向 Sealos 云平台数据库用户的 Web 数据库工作台。**

用户在 Sealos 控制台开通 Postgres / MySQL / MongoDB / Redis / ClickHouse 实例后，点击"连接"即可免配置、免安装、免注册地进入 DataFlow，直接做三类事：

1. **浏览和维护数据**——像传统 GUI 客户端一样查看库、表、集合、键；增删改记录；做表结构变更；导入导出数据。
2. **写查询和命令**——在编辑器里写 SQL、Mongo shell 或 Redis 命令，多语句一次执行，结果分条呈现。
3. **把查询结果沉淀成仪表盘**——把一条查询的结果一键变成图表，挂到可复用、可自动刷新的仪表盘上。

一句话总结：**"一个从 Sealos 点进来就能用的数据库 GUI + 轻量 BI 面板"**。

## 1. Executive Summary

### 1.1 Problem

Sealos 用户开通数据库实例后，目前缺少**一个和云控制台无缝衔接**的 Web 工作台。主流替代品都有一至多项关键缺陷：

- **传统桌面客户端**（DBeaver / TablePlus / Redis Insight 等）：需安装、需手工输入连接串、凭证管理是用户自己的事。
- **各家云厂商的内嵌控制台**：通常只覆盖一两种数据源，能力各自为政，跨 Postgres / MySQL / MongoDB / Redis / ClickHouse 时体验断裂。
- **开源 Web GUI**：多数面向"多连接管理员"设计，缺少"单实例、一点就进、开箱即用"的云端嵌入体验；且几乎都不带仪表盘。

结果是：工程师日常要在多个工具之间来回切换——在 DB 客户端里查数据，在 Grafana/Metabase 里做看板，在 Sealos 控制台里管实例。

### 1.2 Solution

DataFlow 把"**数据库 GUI**"与"**轻量仪表盘**"合并成一个产品，并在以下 4 个维度做了差异化选择：

1. **云原生的零配置登录**——从 Sealos 控制台的"连接"按钮点击进入后，完成身份验证、选定实例、加载元数据、打开工作区全部自动完成，用户不需要输入任何连接信息。
2. **一套 UI 覆盖五种异构数据源**——关系型（Postgres / MySQL / ClickHouse）与非关系型（MongoDB / Redis）共享同一套交互骨架（对象树、Tab、分页、筛选、导出、图表入口），但针对各数据源的数据模型差异提供**独立的编辑管线和命令拦截规则**。
3. **"预览 → 确认 → 执行"的安全编辑管线**——任何会写入数据库的修改（行编辑、结构变更、批量导出等）都在提交前展示将要发生的具体操作，用户确认后才执行；不会默写入、不会静默失败。
4. **查询与仪表盘共生**——任何一次 SQL/Mongo/Redis 查询的结果，都可以一键进入"创建图表"流程，选择图表类型和轴即可挂到仪表盘；仪表盘支持手动或每分钟自动刷新，并缓存最近一次结果，保证再次打开时秒级可见。

### 1.3 Success Criteria（可落地的度量方案）

以下 5 项指标目前尚未在线采集，本节给出**如何度量 + 建议阈值**，可直接作为产品埋点需求落地：

| # | 指标 | 度量方式 | 建议阈值 |
|---|---|---|---|
| M1 | **首次可交互时间 (TTI)** —— 从 Sealos 点"连接"到 DataFlow 工作区侧边栏可点击的时长 | 前端埋点：跳转进入 → 侧边树首次渲染 | P95 ≤ 2.5s |
| M2 | **查询执行往返时延** —— 从用户点 Run 到查询结果首行可见 | 前端埋点：Run 按钮点击 → 结果表首行渲染 | P95 ≤ 800ms（10k 行以内） |
| M3 | **编辑提交成功率** —— 表/集合/Key 编辑的成功提交数 / 总提交数 | 前端埋点：预览确认 → 后端确认返回 | ≥ 98% |
| M4 | **仪表盘复用度** —— 过去 7 天被打开 ≥ 2 次的仪表盘占比 | 前端埋点：打开/切换仪表盘事件 | ≥ 40% |
| M5 | **多语句执行的错误可解释性** —— 多条语句批量执行时，每条失败语句都能独立定位到对应的错误气泡 | QA 自动化断言 / 人工抽检 | 100%（已实现） |

> M1–M4 属于**待埋点**需求；M5 **已在产品中实现**，可直接通过自动化用例验证。

### 2.3 功能柱 ① 登录与连接

**用户故事**

- **US-1.1** 作为 Sealos 用户，点击"连接数据库"后我不需要输入任何凭证，直接进入 DataFlow 并看到该实例的库列表。
- **US-1.2** 作为已登录用户，刷新页面后继续保持登录，不被再次跳转。

**验收标准**

- [x] 支持从 Sealos 控制台带加密凭证的链接直接登录，覆盖 **Postgres / MySQL / MongoDB / Redis / ClickHouse** 五种数据源类型。
- [x] 不支持的数据源类型展示明确错误提示，而非白屏。
- [x] 登录成功后凭证在浏览器本地会话中持久化；刷新页面自动恢复登录态。
- [x] 整个会话只呈现**当前这个实例**的一条连接，不让用户误以为可以切换到其他实例。
- [x] 脱离 Sealos 上下文（例如直接访问 URL）时给出"无可用凭证"的明确错误，不走半成品流程。

**明确不做**

- 多连接管理、连接导入导出、连接测试向导。
- 用户注册、用户组、权限体系（权限由数据库账号本身承担）。
- 从 DataFlow 手工填连接串新建连接（本产品假设用户从 Sealos 跳转进入）。

### 2.4 功能柱 ② 对象树与 Tab 工作区

**用户故事**

- **US-2.1** 作为工程师，左侧提供一棵可展开的对象树，能看到**库 → Schema → 表 / 视图 / 集合 / Key**的完整层级，点击叶节点直接打开数据视图。
- **US-2.2** 作为 DBA，对每类节点右键可获得上下文相关的操作菜单。
- **US-2.3** 作为多任务用户，能同时打开多个表和多个查询编辑器，顶部 Tab 之间自由切换。

#### 2.4.1 对象树层级（按数据源类型自适应）

| 数据源 | 树层级 |
|---|---|
| **Postgres** | `连接 → 数据库 → Schema → Tables 文件夹 → 表` / `Views 文件夹 → 视图` |
| **MySQL / ClickHouse** | `连接 → 数据库 → Tables 文件夹 → 表` / `Views 文件夹 → 视图`（跳过 Schema 层） |
| **MongoDB** | `连接 → 数据库 → 集合` |
| **Redis** | `连接 → 数据库 → Keys 文件夹 → Key` |

#### 2.4.2 通用行为

- [x] **懒加载**：展开父节点时才拉取子节点，大库不会一次性拉完整个元数据。
- [x] **系统对象隐藏开关**：默认隐藏系统库/Schema/对象；右键"Show System Objects"切换。
  - 对 **Postgres**，该开关挂在**数据库节点**上（系统 Schema 属于某个数据库）。
  - 对 **MySQL / MongoDB / ClickHouse**，该开关挂在**连接节点**上（系统库是顶层）。
  - Redis 不涉及系统对象。
- [x] **连接节点**展示对应数据库的品牌图标（MySQL / PostgreSQL / MongoDB / Redis 均有原生 SVG；ClickHouse fallback 到通用数据库图标）。
- [x] **节点颜色区分**：不同节点类型用 5 种色盘区分，便于快速视觉定位。
- [x] **侧边栏宽度**可拖拽调整，范围 180–480px，默认 256px。

#### 2.4.3 右键菜单（按节点类型 × 数据源）

| 菜单对象 | Postgres | MySQL | ClickHouse | MongoDB | Redis |
|---|---|---|---|---|---|
| 连接 | New Query / + New Database / Refresh | New Query / + New Database / Refresh | New Query / + New Database / Refresh | New Query / + New Database / Refresh | New Query / Refresh |
| 数据库 | New Query / + New Table / Export Database / Rename Database / Delete / Refresh | New Query / + New Table / Export Database / Delete / Refresh | New Query / + New Table / Export Database / Delete / Refresh | New Query / + New Collection / Delete / Refresh | New Query / Refresh |
| Schema | New Query / New Table / Refresh | — | — | — | — |
| 表文件夹 | New Table / Refresh | 同左 | 同左 | — | — |
| 视图文件夹 | Refresh | 同左 | 同左 | — | — |
| 表 | Export Data / Duplicate / Design / Rename / Clear / Delete / Refresh | 同左 | Export Data / Clear / Delete / Refresh **（无 Design / Rename / Duplicate）** | — | — |
| 视图 | Export Data / Refresh | 同左 | 同左 | — | — |
| 集合 | — | — | — | Export Collection / Drop Collection / Refresh | — |
| Keys 文件夹 | — | — | — | — | New Key / Refresh |
| Key | — | — | — | — | Export Key / Delete Key / Refresh |

> **ClickHouse 表能力的显著差异**：没有"设计表"、"重命名表"、"复制表"三个入口——因为底层方言的 `ALTER` 语法与 InnoDB/Postgres 差异较大，不在结构化表单的覆盖范围内。ClickHouse 的这些操作用户需要回到查询编辑器手写 DDL。
>
> **系统对象开关**：在存在系统 Schema/库的场景下，右键菜单末尾还会追加 `Show / Hide System Objects` 项——MySQL / MongoDB / ClickHouse 挂在**连接**节点，Postgres 挂在**数据库**节点（参见 2.4.2）。

#### 2.4.4 Tab 工作区

- [x] **四种 Tab 类型**：`查询编辑器 / SQL 表 / Mongo 集合 / Redis Key 详情`。
- [x] **Tab 复用规则**：同一张表/集合/Key 重复点击会复用已有 Tab；**查询编辑器每次都是新 Tab**（分析师经常并行多个草稿）。
- [x] **Tab 右键菜单**：关闭 / 关闭其他 / 关闭全部。
- [x] **写操作触发全树刷新**：写入（DDL/DML）成功后自动刷新对象树的相关节点。
- [x] **活动栏切换**：左侧 `Connections` 与 `Analysis` 两个顶层模式，状态独立保留。

**明确不做**：工作区多 pane 分屏、Tab 拖拽重排序、收藏/固定节点。

### 2.5 功能柱 ③（共通）数据浏览与编辑的通用能力

以下能力在 SQL 表、MongoDB 集合、Redis Key 三种数据视图中**结构相同、交互一致**：

- [x] **分页**：页码跳转、每页行数（默认 50）切换、总数显示、首页/上一页/下一页/末页。
- [x] **行内查找（FindBar）**：`Cmd/Ctrl+F` 打开的全页搜索栏，基于当前页可见数据做**客户端匹配**；支持回车跳到下一个命中、Shift+回车跳到上一个、Esc 关闭、实时高亮并自动滚动到当前命中。
- [x] **导出**：顶部导出按钮，支持 CSV / JSON / SQL / Excel 四种格式；导出过程有进度提示，部分失败会列出失败条目。
- [x] **刷新按钮**：手动强制重新拉取。
- [x] **查询跳转**：顶部"Query"按钮一键打开预填了当前对象访问语句的查询编辑器（SQL 预填 `SELECT * FROM ...`；Mongo 预填 `db.collection.find({})`；Redis 预填空白编辑器，保留当前数据库上下文）。
- [x] **出图入口**：顶部"图表"按钮一键把当前视图的数据喂给图表创建对话框（见 2.12）。
- [x] **加载/错误态**：统一的 loading spinner；错误态附带 Retry 按钮；网络异常不会把已有数据清空。

> 这些是"骨架共通能力"；**每种数据源独有的编辑语义**分别见 2.6 / 2.7 / 2.8。

### 2.6 功能柱 ③a SQL 数据视图（Postgres / MySQL / ClickHouse）

**数据网格**

- [x] 真实表格布局（行号列 + 可隐藏数据列 + 固定表头）。
- [x] **列头**呈现：列名 + `PK` 徽章 + `FK` 徽章 + 简化后的类型（例如 `character varying` → `char`、`timestamp without time zone` → `timestamp`）+ 排序指示 + 列操作下拉菜单 + 列宽拖拽手柄。
- [x] **排序**：任意列升/降序切换；支持"清除排序"。同时只排一列。
- [x] **列可见性**：筛选弹窗里有"可见列"勾选组；隐藏后在空列头显示"{N} 列已隐藏"。
- [x] **列宽拖拽**：拖动列头右侧手柄即可调整；已调整过的列会固定宽度，未调整的列自适应。

**筛选（服务端 WHERE）**

- [x] 筛选弹窗里可添加任意多条字段级条件，以胶囊形式展示在筛选栏，每条可单独删除。
- [x] **支持运算符**：`= / != / <> / > / < / >= / <= / LIKE / NOT LIKE / IN / NOT IN / IS / IS NOT / IS NULL / IS NOT NULL`（最后两个不需要填值）。
- [x] 筛选条件与排序组合时都推送到后端执行，不是前端过滤。

**编辑（Changeset 管线）**

- [x] **未提交修改集**：所有 insert / update / delete 先收集在本地；UI 以颜色差异（蓝：新增、绿：更新、红：删除）标记未提交变更。
- [x] **主键保护**：主键列只读（无论是否有 FK 约束），但**新增行可编辑所有列**。
- [x] **Undo 栈**：`Cmd/Ctrl+Z` 逐步撤销本地修改；Undo 按钮禁用状态同步。
- [x] **批量标记删除**：多选行后点击"Mark selected for delete"整批转为删除状态。
- [x] **预览对话框**（Preview Pending Changes）：弹出展示将要执行的 SQL 语句清单 + "{updates} 行修改 / {inserts} 行插入 / {deletes} 行删除"摘要。**SQL 仅用于显示，不可手动编辑**。
- [x] **提交对话框**（Submit Changes）：含二次确认的摘要；提交后若部分失败，成功的变更从 pending 集中移除、失败的保留，继续显示错误详情。
- [x] **只读模式**：后端返回 `disableUpdate` 标记时（视图或不可写存储），隐藏所有编辑按钮。

**安全保护**

- [x] **丢弃确认**：有未提交修改时切换页码、排序、筛选、列可见性、刷新等任何会重新拉数据的动作，会弹出"Discard pending changes?"对话框，用户必须显式确认或取消。
- [x] **关闭拦截**：有未提交修改时关闭浏览器标签/刷新页面会触发 `beforeunload` 浏览器原生确认。
- [x] **删除行二次确认**：单行删除需要**键入表名字面量**才能启用删除按钮。
- [x] **表切换重置**：切到另一张表时自动重置所有本地状态（筛选、排序、搜索、可见列、未提交修改）——不会串用。

### 2.7 功能柱 ③b MongoDB 集合视图

**文档展示**

- [x] 文档以**卡片列表**形式展示（不是表格），每张卡片是一个 pretty-printed 的 JSON（去掉外层大括号以节省空间）。
- [x] 卡片支持双击打开完整文档编辑对话框。
- [x] 改动态以卡片背景色区分：**蓝 = 新增、绿 = 修改、红 = 删除（带删除线）**。
- [x] 空集合态展示"No documents found"占位文案。

**文档级操作**

- [x] **Add Document** 模态：内嵌 JSON 编辑器，空文档（无字段）被拒绝；新增文档显示在列表顶部的"pending inserts"区。
- [x] **Edit Document** 模态：展示原始 JSON 并允许整体替换。
- [x] **批量删除**：勾选多个卡片后一次标记删除。
- [x] **Undo**：逐步回退最近一次 changeset 修改。

**Changeset 管线与安全保护**

- [x] 与 SQL 表结构一致的 **未提交修改集 + 预览 + 提交 + 部分失败恢复** 流水线。
- [x] 预览对话框用 **mongo shell 命令**形式展示（`db.collection.updateOne({...}, {...})` / `insertOne(...)` / `deleteOne(...)`），方便 DBA 复制到 mongosh 复核。
- [x] **丢弃确认**：有未提交修改时切换分页、筛选、搜索词会弹出"Discard pending changes?"。
- [x] **关闭拦截**：`beforeunload` 浏览器原生确认。

**筛选（服务端）**

- [x] **字段发现**：从当前页最多 50 个文档里提取所有顶层字段名，作为筛选字段候选。
- [x] **每个字段只能出现一次**（防止同字段冲突条件）；新增条件时自动排除已使用的字段。
- [x] **支持运算符（8 种）**：`$eq` / `$ne` / `$regex`（显示为"Contains"）/ `$gt` / `$lt` / `$gte` / `$lte` / `$in`（逗号分隔）。
- [x] **不支持嵌套/分组条件**：如果外部传入了嵌套条件会给出 "Only flat atomic filters are supported" 的提示并忽略嵌套部分。

**导出**

- [x] 独立的 `Export Collection` 对话框：支持 CSV / JSON / SQL / Excel 格式、可选 MongoDB 筛选表达式（例 `{ "status": "active" }`）、可选行数上限。

### 2.8 功能柱 ③c Redis Key 视图

**类型自动识别与专用渲染**

- [x] 打开一个 Key 时根据返回列自动识别类型：
  - 含 `field` → **Hash**
  - 含 `member` → **Sorted Set**
  - 含 `index` + 可更新 → **List**
  - 含 `index` + 只读 → **Set**
  - 其他 → **String**
- [x] 渲染为同一张数据表格，列组合随类型变化（例如 Hash 展示 `field / value`、ZSet 展示 `member / score`、List 展示 `index / value`）。

**行内编辑**

- [x] **点击单元格进入编辑态**：Enter 提交、Esc 取消、Tab 提交并跳到下一列。
- [x] **哪些列可编辑**：
  - `index`（List/Set）不可编辑。
  - `field`（Hash）和 `member`（ZSet）不可直接改名（属于 Key 标识部分）。
  - 其他列均可直接修改。
- [x] **ZSet score 特殊处理**：修改 score 时直接调用 ZADD 更新该 member 的分值。
- [x] **错误处理**：提交失败时顶部红色横幅展示错误，数据保留。

**新增与删除**

- [x] **Add Entry**：在顶部加一行可填字段的空行，支持 Enter 提交 / Esc 取消；按类型预填对应字段（例如 Hash 需填 `field` + `value`，ZSet 需填 `member` + `score` 默认 `0`）。
- [x] **String 类型不可新增 entry**（整个 Key 就是一个值），只能在 Key 层面编辑。
- [x] **批量删除**：多选行后点击删除，**List 类型按 index 从大到小逐条删除**（避免 index 漂移）。
- [x] **删除前确认对话框**：显示"删除 {N} 条"的二次确认。

**排序**

- [x] 任意列升/降序切换；同时只排一列。

**创建 Key（New Key 模态）**

- [x] 5 种类型**全部可新建**，每种类型有专用表单子编辑器：
  - **String**：单行值输入。
  - **Hash**：可增减的 `field / value` 表。
  - **List**：可增减的值列表。
  - **Set**：可增减的 member 列表。
  - **ZSet**：可增减的 `score / member` 表（score 数字输入）。
- [x] Key 名称必填；至少填一个值才能提交。

**编辑 Key（Edit Key 模态）**

- [x] **只支持 String 类型**的整体替换编辑；其他类型要求用户回到**行内编辑**（详情视图）完成。
- [x] 进入 Edit Key 模态时 Key 名称与类型只读。

**导出 / 删除 Key**

- [x] 独立的 `Export Key` 对话框：CSV / JSON / SQL / Excel；可选行数上限。
- [x] `Delete Key` 对话框：**键入 Key 名字面量**才能启用删除按钮。

**Redis 视图没有的能力**（和 SQL/MongoDB 的差异）

- [x] **没有筛选（FilterBar）**：Redis Key 详情视图只有 FindBar 客户端搜索，没有服务端字段级筛选。
- [x] **没有 changeset / 未提交修改集**：Redis 的编辑是**即时提交**（inline commit），无预览对话框、无 Undo 栈、无 `beforeunload` 拦截。
- [x] **没有列可见性控制**。

### 2.9 功能柱 ④ 结构变更（DDL）

**用户故事**

- **US-4.1** 作为 DBA，我希望通过**表单**而非手写 SQL 就能完成常见的库/表/列/索引/外键建改。
- **US-4.2** 作为 DBA，在执行有风险的结构变更之前我希望能看到将要执行的语句。

**数据库级操作（按数据源）**

| 操作 | Postgres | MySQL | ClickHouse | MongoDB | Redis |
|---|---|---|---|---|---|
| 创建数据库 | ✅ `CREATE DATABASE` | ✅ | ✅ | ⚠️ 需先创建第一个集合 | ❌ |
| 重命名数据库 | ✅ `ALTER DATABASE RENAME` | ❌ | ❌ | ❌ | ❌ |
| 删除数据库 | ✅ `DROP DATABASE` | ✅ | ✅ | ✅ `dropDatabase()` | ❌ |
| 整库导出（ZIP） | ✅（按 schema 分子目录） | ✅ | ✅ | ❌ | ❌ |

> 不支持的操作（如 MongoDB 重命名库、ClickHouse 重命名库 via UI）**都会返回一条明确的错误信息**，而不是静默失败或按钮消失。

**表/集合/Key 级别的结构化表单**

**SQL 数据源的"建表"表单**（Create Table Modal）：

- [x] 字段：表名 + 列列表。
- [x] 每列：`name` + `type`（从预设类型表选，含 INT / BIGINT / VARCHAR(...) / TEXT / DECIMAL / DATE / DATETIME / TIMESTAMP / JSON / BLOB 等 22+ 个常用类型）+ `PK` 勾选 + `Null` 勾选。
- [x] 默认预填一行 `id INT PRIMARY KEY NOT NULL`。
- [x] 可任意增删列。

**SQL 数据源的"改表"表单**（Edit Table Modal，MySQL / Postgres）：

- [x] 三个 Tab：**Fields / Indexes / Foreign Keys**。
- [x] **Fields Tab**：列出所有现有列，可新增、修改类型/可空性、标记删除（hover 出现删除按钮）。
- [x] **Indexes Tab**：新增/修改/删除索引，字段用 MultiSelect 选列、可勾选 Unique。
- [x] **Foreign Keys Tab**：新增/修改/删除外键，含 `name / column / ref table / ref column / ON DELETE / ON UPDATE`。
- [x] 表单校验：列名必填、索引名必填、索引至少选一列、外键所有字段必填。
- [x] 提交后显示"X 变更成功、Y 变更失败"的分项结果。
- [x] **ClickHouse 不支持 Edit Table**（只能新建，不能改）。

**清空表**（Clear Data，SQL 专有）：

- [x] 两种模式二选一：
  - **Fast (TRUNCATE)**：重置自增计数、跳过行级触发器、大表最快。
  - **Safe (DELETE)**：保留自增计数、触发行级触发器、大表较慢。
  - ClickHouse 的 `DELETE` 模式实际翻译为 `ALTER TABLE ... DELETE WHERE 1=1`（ClickHouse 特有语法）。

**复制表**（Duplicate Table，仅 MySQL / Postgres）：

- [x] 可选 **Structure Only** 或 **Structure and Data** 两种模式。
- [x] 方言自动适配：MySQL 用 `CREATE TABLE x LIKE y`；Postgres 用 `CREATE TABLE x (LIKE y INCLUDING ALL)`。

**重命名表**（Rename Table，仅 MySQL / Postgres）：

- [x] 简单表单：当前名（只读）+ 新名。

**删除表 / 删除 Key / Drop Collection**：

- [x] 所有破坏性操作都要求**用户键入对象名字面量**才能启用删除按钮（防手滑）。
- [x] 弹窗展示"此操作不可撤销"警告文案。

**MongoDB 集合操作**：

- [x] `Create Collection`：表单填集合名。
- [x] `Drop Collection`：键入名字面量确认。

**Redis Key 操作**：

- [x] 详见 2.8 功能柱 ③c 的 "Create Key" 与 "Delete Key" 段落。

**明确不做**

- ER 图 / 可视化关系图。
- DDL 变更历史 / 版本对比。
- 数据库备份与恢复。
- ClickHouse 的 Edit Table（当前走手写 SQL）。

### 2.12 功能柱 ⑤b MongoDB 命令编辑器

### 2.13 功能柱 ⑤c Redis 命令编辑器

**语言模式**

- [x] **纯文本**（Monaco 的 `plaintext` 模式，无高亮也无补全）。

**多语句拆分**

- [x] **按换行拆分**：每行一条命令。
- [x] **注释行**：以 `#` 或 `//` 开头的整行被视为注释并丢弃（避免复制教程时把示例说明误发到 Redis）。

**安全拦截：不支持的命令**

Redis 编辑器以"每条命令一个独立连接"的模型工作，因此对"依赖连接状态"的命令在执行前就拦截并中止整批，错误信息解释原因：

| 被拦截的命令 | 拦截原因 |
|---|---|
| `MULTI` / `EXEC` / `DISCARD` | 事务依赖同一连接的状态 |
| `WATCH` / `UNWATCH` | 乐观锁同样依赖连接状态 |
| `SUBSCRIBE` / `PSUBSCRIBE` / `UNSUBSCRIBE` / `PUNSUBSCRIBE` | Pub/Sub 会阻塞连接 |
| `MONITOR` | 长阻塞命令，阻塞连接 |

**读操作识别**

- [x] 识别为"读"的命令前缀（29 个）：`GET / MGET / HGET / HGETALL / HMGET / HKEYS / HVALS / HLEN / LRANGE / LLEN / LINDEX / SMEMBERS / SCARD / SISMEMBER / ZRANGE / ZREVRANGE / ZRANGEBYSCORE / ZSCORE / ZCARD / KEYS / SCAN / EXISTS / TYPE / TTL / PTTL / DBSIZE / INFO / STRLEN / PING / ECHO / CONFIG`。
- [x] 其他命令被视为写操作，成功后触发对象树刷新。

### 3.2 交付方式

- **嵌入式 Web 应用**：用户通过 Sealos 控制台跳转进入，无需本地安装。
- **单实例会话**：每次会话绑定一个数据库实例；切换实例需回到 Sealos 控制台重新进入。
- **浏览器兼容**：面向现代浏览器（Chromium / Firefox / Safari 最新版）。
- **零运维**：产品作为 Sealos 套件的一部分部署，用户侧没有任何运维成本。

### 3.3 安全与隐私（用户视角）

- **凭证在传输中加密**：从 Sealos 控制台跳转时，凭证以加密形式附在 URL 上，不会以明文出现在浏览器历史或日志里。
- **凭证只保存在用户本地浏览器**：不集中存储；关闭浏览器标签后可主动清除。
- **写操作强制二次确认**：
  - SQL 表 / MongoDB 集合：Changeset 预览 + 提交双对话框。
  - 结构变更：表单提交前展示将要执行的语句。
  - 破坏性操作（Delete Row / Delete Table / Drop Collection / Delete Key / Delete Database）：要求**键入对象名字面量**才能启用确认按钮。
- **危险命令前置拦截**：Redis 订阅/事务、SQL 独立事务控制、MongoDB 非命令语句都在前端执行前拦截。
- **关闭拦截**：SQL 表和 MongoDB 集合在有未提交修改时触发浏览器原生 `beforeunload` 确认。
- **权限由数据库账号决定**：DataFlow 不重新设计权限体系；用户能做什么完全取决于 Sealos 下发的数据库账号权限。

### 3.4 已知的安全边界（产品侧诚实披露）

- 浏览器本地存储的凭证为**明文**。任何能在该浏览器执行脚本的攻击者（XSS）都有可能读取；建议不在共享机器上长时间保留会话。
- 仪表盘绑定的查询**不区分只读/可写**：带写操作的查询会在仪表盘刷新时被反复执行。建议团队通过**数据库账号本身的权限**限制仪表盘作者的可执行范围。
- 产品本身不内置审计日志。用户操作的可追溯性依赖数据库实例层面的审计能力。

## 4. 当前限制与已知风险

> 下列条目描述**今天用户可能感知到的约束**，供产品侧在向客户介绍时使用，不是工程 backlog。

### 4.1 性能/规模类

| # | 限制 | 具体表现 |
|---|---|---|
| R1 | **行内查找是前端匹配** | `Cmd/Ctrl+F` 只在**当前页可见数据**内匹配，不跨分页、不走后端；因此想"全表搜一个字符串"需要用筛选条件或查询编辑器。 |
| R2 | **大 schema 补全首屏稍慢** | 打开 SQL 编辑器时会一次性拉该 schema 下所有表的列信息用于补全；含数百张表的 schema 首次加载需要几秒（加载后缓存）。 |
| R3 | **仪表盘一次性并发刷新** | 含很多图表的仪表盘在点 Refresh 时会同时触发所有图表的查询，可能短时压满数据源。 |
| R4 | **首屏资源体积较大** | 首屏加载包含完整代码编辑器静态资源，进入"仅查看仪表盘"的用户也会加载。 |
| R5 | **MongoDB 字段发现是采样** | 筛选器候选字段从**当前页最多 50 个文档**里采样，schema-less 数据里不常见的字段可能不在下拉列表里。 |

### 4.2 功能/体验类

| # | 限制 | 具体表现 |
|---|---|---|
| R6 | **单实例假设** | 每次会话只绑定一个数据库实例；不能并行操作两个实例。 |
| R7 | **查询不持久化** | 查询编辑器的文本在 Tab 关闭后丢失，没有查询历史或收藏。 |
| R8 | **Redis 编辑没有预览 / Undo** | Redis Key 的行内编辑是即时提交，没有 changeset 预览、没有 Undo、没有关闭拦截——和 SQL/Mongo 的编辑体验不对称。 |
| R9 | **Redis 详情视图没有筛选** | 只有客户端 FindBar，没有服务端字段级筛选（与 SQL/Mongo 不对称）。 |
| R10 | **ClickHouse 无"改表"能力** | Design / Rename / Duplicate 三个入口在 ClickHouse 上不可用（对应方言 ALTER 语法不在结构化表单覆盖范围内），需要回到查询编辑器手写 DDL。 |
| R11 | **MongoDB 无"重命名库"** | 底层不支持，DataFlow 返回明确错误。 |
| R12 | **MongoDB 无整库导出** | 只能逐集合导出。 |
| R13 | **Redis 不能通过 UI 建数据库** | Redis 的逻辑库（0–15）是固定的，不能新建。 |
| R14 | **Redis Edit Key 模态只支持 String** | 其他类型只能通过详情视图的行内编辑修改。 |
| R15 | **仪表盘 widget 只有图表一种** | 没有表格、文本、图片、单值卡片、过滤器等组件。 |
| R16 | **图表类型仅 4 种** | 柱 / 线 / 饼 / 面积；没有散点、雷达、地图等高级可视化。 |
| R17 | **刷新频率只有两档** | 按需 or 每分钟；没有 5/15/30 分钟或 cron。 |
| R18 | **MongoDB 编辑器无补全** | SQL 编辑器基于 schema 元数据做补全，但 MongoDB 编辑器没有等价能力。 |

### 4.3 生态类

| # | 限制 | 具体表现 |
|---|---|---|
| R19 | **脱离 Sealos 无法新建连接** | 产品当前只走 Sealos 跳转入口，不提供"手动填连接串"的备用登录方式。 |
| R20 | **无数据导入能力（从文件）** | 产品支持导出为 CSV / JSON / SQL / Excel，但没有"上传 CSV 建表并导入"的反向能力。 |
| R21 | **无 AI 辅助** | 产品当前不提供自然语言生成 SQL / 解释 SQL / 数据洞察等 AI 能力。 |

## 5. 下阶段候选计划

> **性质**：以下为**候选项 (candidates)**，**非承诺清单**。不含时间线、不含版本号、不含资源分配。列出目的是为产品决策会议提供一张"可以讨论的菜单"。
> **分类依据**：按"落地路径"分为四类——A 类是产品后台已具备能力但 UI 未接入；B–D 类是全新增量。
> **与 §4 的关系**：标注 `(→ R##)` 的条目对应 §4 中的一条已知限制。

### 5.1 A 类 · 激活后台已具备的能力

此类候选的共同特征是：**核心数据通路已经存在**，用户侧只欠一套工作流和交互。因此落地成本显著低于 B–D 类，是"投入产出比"最高的一档。

| # | 候选 | 用户价值 | 备注 |
|---|---|---|---|
| **A1** | **AI 对话查询** | 自然语言 → SQL；SQL 解释；查询结果一句话洞察 | 覆盖多个 LLM 供应商；对数据分析师与非 DBA 工程师价值最高 |
| **A2** | **文件导入**（CSV / Excel / SQL 脚本） | 补齐"只导出不导入"的对称缺口；支持字段映射预览、追加/覆盖/Upsert 三种模式 | 对应 §4 的 **R20** |
| **A3** | **Mock 数据生成** | 一键为表生成测试数据；能沿外键依赖生成一致数据 | 面向 Dev/Test 场景；可与 A2 联动（生成后导出为种子文件）|
| **A4** | **表关系图** | 以节点-边图形式浏览表之间的外键关系 | 大 schema 下的"探索式浏览"体验补强 |
| **A5** | **查询建议** | 打开一个 schema 时自动推荐常用查询模板 | 降低新用户的冷启动成本 |

### 5.2 B 类 · 填补用户可见的体验缺口

此类候选直接对应 §4 中的已知限制，属于"已知痛点修复"。

| # | 候选 | 对应限制 | 说明 |
|---|---|---|---|
| **B1** | 查询历史与收藏 | → R7 | 关 Tab 不丢 SQL；常用查询可 pin |
| **B2** | 更多 widget 类型 | → R15 | 优先级：**表格 widget** > 单值卡片 > 过滤器 > 文本/图片 |
| **B3** | 更多图表类型 | → R16 | 散点 / 雷达 / 热力图 / 地图等 |
| **B4** | 自定义刷新频率 | → R17 | 5 / 15 / 30 分钟粒度；或 cron 表达式 |
| **B5** | Redis 编辑的 changeset 化 | → R8 | 给 Redis Key 详情视图加上和 SQL/Mongo 对称的预览 + 提交 + Undo + 关闭拦截流水线 |
| **B6** | Redis 详情服务端筛选 | → R9 | 给 Redis Key 视图加字段级筛选（类似 Hash 字段、ZSet score 范围等） |
| **B7** | ClickHouse 高级 DDL | → R10 | 补齐复杂 `ALTER TABLE` 语法的结构化表单 |
| **B8** | MongoDB 整库导出 | → R12 | 多集合一次打包 |
| **B9** | MongoDB 编辑器补全 | → R18 | 基于集合采样字段做补全 |
| **B10** | Tab 拖拽重排序 | — | 小改动、高频使用 |

### 5.3 C 类 · 性能与规模

面向"数据量变大后才暴露"的可用性瓶颈。

| # | 候选 | 对应限制 | 说明 |
|---|---|---|---|
| **C1** | 行内查找改为服务端全表搜索 | → R1 | 大表（>1 万行）可用性的硬需求 |
| **C2** | Schema 补全元数据按需加载 | → R2 | 大 schema 首屏不再等几秒 |
| **C3** | 仪表盘刷新限流 / 串行化 | → R3 | 含 ≥10 个图表的仪表盘不会瞬时打满数据源 |
| **C4** | 编辑器资源按需加载 | → R4 | 首屏体积下降，非编辑器页面更快可交互 |
| **C5** | MongoDB 筛选字段全库发现 | → R5 | 不再从 50 个文档里采样，而是从完整 schema 元数据拿 |

### 5.4 D 类 · 生态扩展（战略级方向决策）

此类候选涉及产品定位变化，落地前需要先做方向性决策——**不是"做或不做"的工程问题，而是"产品是否愿意走这一步"的商业问题**。

| # | 候选 | 说明 |
|---|---|---|
| **D1** | **手动连接串登录**（→ R19） | 脱离 Sealos 上下文独立可用；对开源用户友好，但会模糊"Sealos 嵌入式工作台"的产品定位 |
| **D2** | **多实例会话**（→ R6） | 同一标签页并行操作两个以上实例；对 DBA 价值高，但需要重构对象树与 Tab 体系 |
| **D3** | **审计日志** | 记录谁在什么时间对什么对象做了什么写操作；企业客户刚需，也是"安全边界"的关键补强（见 §3.4）|
| **D4** | **仪表盘查询的只读性标记** | 在保存图表时识别查询是否包含 DML/DDL 并强制二次确认；补 §3.4 第二条风险 |
| **D5** | **仪表盘分享与权限** | 仪表盘链接分享、只读访问、看板级权限；配合 D3 形成企业版基础 |

### 5.5 推荐优先级 Top 5

在不定时间线的前提下，如果按"**落地成本 × 用户价值**"综合权衡，建议下阶段优先评估：

| 排序 | 候选 | 理由 |
|---|---|---|
| ⭐⭐⭐⭐⭐ | **A2 文件导入** | 后端已就绪 + 对称缺口 + 客户高频询问 |
| ⭐⭐⭐⭐⭐ | **A1 AI 对话查询** | 后端已就绪 + 产品差异化最强的一张牌 |
| ⭐⭐⭐⭐ | **B2 表格 widget** | 仪表盘场景中最高频的"不是图表但需要的东西" |
| ⭐⭐⭐⭐ | **B1 查询历史** | 低成本小痛点，每天被用户感知 |
| ⭐⭐⭐ | **B5 Redis 编辑 changeset 化** | 消除三种数据源的编辑体验不对称性，对 Redis 重度用户价值明显 |

### 5.6 未纳入候选的方向

以下方向**明确不在当前考虑范围内**，列出是为了让决策讨论有完整的"我们不做 X"上下文：

- **多租户 SaaS 化**（把 DataFlow 从 Sealos 内嵌组件改为独立 SaaS 产品）——产品定位冲突。
- **OLAP 建模层 / 语义层**（dbt-like）——与仪表盘的轻量定位不符，应交给专业 BI 工具。
- **代码协作**（类似 Google Docs 的多人同时编辑 SQL）——受众过窄。
- **Notebook 化工作区**（类似 Hex / Mode）——与当前"Tab + 仪表盘"的交互范式冲突，若要走这条路需要产品形态级重构。

## 6. 术语表

- **工作区 (Workspace)**：DataFlow 的主内容区，包含对象树、Tab 栏和当前 Tab 的视图。
- **活动区 (Activity Bar)**：最左侧的图标栏，用于在 Connections（数据库操作）和 Analysis（仪表盘）两种模式间切换。
- **对象树 (Object Tree)**：左侧用于浏览数据库对象层级的可展开树。
- **Tab**：工作区顶部的标签，一个 Tab 对应一个查询编辑器、一张表、一个集合或一个 Redis Key。
- **数据视图 (Data View)**：SQL 表 / MongoDB 集合 / Redis Key 三种展开视图的统称。
- **Changeset / 未提交修改集**：SQL 表 / MongoDB 集合的本地未提交编辑集合，可在一次预览中统一提交或放弃。
- **Changeset 预览**：提交未提交修改前展示"将要发生什么"的对话框；本产品的安全核心机制之一。
- **丢弃确认 (Discard Guard)**：有未提交修改时切换分页/排序/筛选等操作前弹出的确认对话框。
- **关闭拦截 (beforeunload)**：有未提交修改时关闭标签/刷新浏览器触发的原生确认。
- **仪表盘 (Dashboard)**：由一到多个图表组成的可视化看板，可被反复打开、自动刷新。
- **图表 (Widget / Chart)**：仪表盘上的一个可视化单元；当前等同于"由一条查询驱动的图表"。
- **快照 (Snapshot)**：图表最近一次执行的结果缓存，保证再次打开仪表盘时可以秒级显示。
- **Stale 标记**：快照仍在但新一轮刷新正在进行时的状态提示。
- **Find Bar**：基于 `Cmd/Ctrl+F` 的客户端查找栏，匹配当前页可见数据。
- **Filter Bar / Filter Modal**：基于字段级条件的服务端筛选（SQL / MongoDB 有；Redis 无）。
